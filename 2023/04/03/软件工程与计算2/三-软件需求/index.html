软工2-三-软件需求

单纯的软件系统是不能解决问题的，只有和现实世界之间形成有效互动才能实现问题的解决。

第三部分 软件需求

一、需求工程

需求工程的概念

所有需求处理活动的总和。它收集信息、分析问题、整合观点、记录需求并验证其正确性，最终描述出软件被应用后与其环境互动形成的期望效应。

三个主要任务：

  • 需求工程必须说明软件系统将被应用的应用环境及其目标，说明用来达成这些目标的软件行动（为什么、做什么）

  • 需求工程必须将目标和功能反映到软件系统中，映射为可行的软件行为，并对软件行为进行准确的规格说明

  • 现实世界是不断变化的世界，因此需求工程还需要妥善处理目标和功能随着时间演化的变动情况

需求工程的活动

  • 需求开发
    • 需求获取
    • 需求分析
    • 需求规格说明
    • 需求验证
  • 需求管理

1. 需求获取

从人、文档或者环境中利用各种方法和技术获取需求

  • 目标分析：根据问题确定目标、分析利害关系人确定目标
  • 常见困难：用户与开发人员的认知差异
  • 获取方法：面谈、问卷、文档分析、头脑风暴、专题讨论、原型

2. 需求分析

  1. 通过建模整合各种信息
  2. 为问题定义出一个需求集合，该集合能界定一个问题的解决方案
  3. 检查需求中的错漏，加以修正
(1) 边界分析
  1. 定义项目的范围
  2. 系统边界的定义要保证系统能和周围环境形成有效互动
  3. 系统用例图常被用于定义系统边界
(2) 需求建模

为展现和解释信息而进行的抽象描述活动

  • 常用类图、顺序图、状态图

3. 需求规格说明

在系统用户间交流需求信息

  • 要简洁、精确、一致和易于理解

  • 需求工程师在该阶段的重要工作：定制文档模版、编写文档

4. 需求验证

需求规格说明文档的要求：

  1. 每条需求都正确、准确地反映了用户意图
  2. 需求集在整体上具有整体性和一致性
  3. 文档的组织方式和需求的书写方式具有可读性和可修改性

验证的方法：同级评审、原型、模拟

需求管理

  1. 保证需求作用的持续、稳定和有效发挥
    • 在需求开发活动后，设计、测试、实现等后续的软件系统开发活动都需要以围绕需求开展工作
  2. 进行变更控制
    • 纳入实现合理的变更需求、拒绝不合理变更需求，控制变更成本和影响范围

二、需求基础

需求的定义

  • 用户为了解决问题或达到某种目标所需要的条件或能力
  • 系统或系统部件为了满足合同、标准、规范或其他正式文档所规定的要求而需要具备的条件或能力
  • 对上述中的一个条件或一种能力的一种文档化表述

需求是一种解决问题后所能达到的期望。

问题域：

  • 现实世界运行规律的一种反映
  • 需求的产生地与解决地
  • 最终的软件产品要在现实中部署，它能够部分影响问题域，但不能任意改变现实
    • 软件开发必须尊重问题域，不能因为技术原因妄自修改现实情况

问题的解决：

  • 基础：模拟与共享现象
  • 方法：直接、间接
  • 解决方案：需求规格说明

需求开发的目标

将现实世界问题域描述转化为计算世界的规格说明。

规格说明：软件产品的方案描述，以软件产品的运行机制为主要内容。

  • 关注对外交互的方式描述软件解决方案

需求层次性

  • 业务需求：系统具备的特性

  • 用户需求：系统能帮助用户做些什么

  • 系统需求：用户对系统行为的期望

需求分类

需求谱系

  • 需求
    • 项目需求
    • 过程需求
    • 系统需求
      • 软件需求
      • 硬件需求
      • 其他需求
  • 不切实际的期望

需求分类

  • 功能需求
    • 不考虑物理约束的情况下，用户希望系统能够执行的活动
    • 主要表现为系统和环境之间的行为交互
  • 性能需求
    • 系统整体或系统组成部分应该拥有的性能特征
  • 质量属性
    • 系统完成工作的质量
  • 对外接口
    • 系统和环境中其他系统需要建立的接口
  • 约束
    • 进行系统构造时需要遵守的约束

三、需求分析

需求分析的任务

  • 建立分析模型，达成开发者和用户对需求信息的共同理解
  • 依据共同的理解发挥创造性，创建软件系统解决方案

需求分析模型

模型是对事物的抽象，帮助人们在创建一个事物之前可以有更好的理解。

面向对象方法模型

模型 描述
用例图 描述用户与系统的交互。从交互的角度说明了系统的边界和功能范围。
类图 描述应用领域中重要的概念以及概念之间的关系。它捕获了系统的静态结构。
顺序图 描述系统中一次交互的行为过程，说明了在交互中的对象协作关系。
状态图 描述系统、用例或者对象在其整个生命期内的状态变化和行为过程。

结构化方法模型

模型 描述
数据流图 从数据传递和加工的角度，描述了系统从输入到输出的功能处理过程。运用功能分解的方法，用层次结构简化处理复杂的问题。
实体关系图 描述系统中的数据对象及其关系，定义了系统中使用、处理和产生的所有数据。

建模

建立模型的过程被称为建模。

  • 建模是对系统进行思考和推理的一种方式

  • 建模的目标是建立系统的一个表示，这个表示以精确一致的方式描述系统，使得系统的使用更加容易

  • 建模的常用手段：抽象和分解

四、面向对象分析

简单过程

  • 系统用例图——细化系统的对外交互——>用例模型/用例描述

  • 用例模型/用例描述——明确用例中的协作对象——>概念类图/领域模型

  • 用例模型/用例描述——明确用例中的协作行为——>顺序图、状态图、对象约束语言

1. 用例图

用例的定义

在系统（或者子系统或者类）和外部对象的交互当中所执行的行为序列的描述，包括各种不同的序列和错误的序列，它们能够联合提供一种有价值的服务。

用例图的基本元素

参与者、用例、关系、系统边界

  • 参与者（小人形）
    • 一个与正在开发的系统进行交互的用户或系统所扮演的角色
    • 用例图中的一个参与者可以代表多个用户或系统、扮演多种角色
  • 用例（椭圆）
  • 关系（不同的线和箭头及上面的补充文字）
  • 系统边界（矩形）
    • 参与者总在系统边界外

用例图的建立

  1. 目标分析与解决方向的确定
  2. 寻找参与者
  3. 寻找用例
  4. 细化用例
  • 如果用例的粒度不合适就需要进行细化和调整
    • 判断标准：用例描述了为应对一个业务事件，由一个用户发起，并在一个连续时间段内完成，可以增加业务价值的任务。

常见错误

  • 不要把用例细化为单个操作
    • 例：不要将用户管理细化为增加、修改和删除三个更小的用例，因为它们要联合起来才能体现出业务价值。
  • 不要把同一个业务目标细化为不同用例
    • 例：特价策略制定和赠送策略制定。
  • 不要把没有业务价值的内容作为用例
    • 例：“登录”(应该描述为安全性质量需求)、“数据验证”(应该描述为数据需求)、“连接数据库”(属性软件内部实现而不是需求)等。

2. 概念类图

概念类图又被称为领域模型。

类图是面向对象分析方法的核心，描述类和类之间的关系。

  • 和设计类图的不同：关注系统与外界的交互，而不是软件系统的内部构造机制
  • 类型、方法、可见性等复杂的软件构造细节不会在概念类图中

概念类图的基本元素

对象、类、链接、关联、继承

继承、组合、聚合、关联的图示：

classDiagram 
	A <|-- B
	C *-- D
	E o-- F
	G -- H

建立概念类图

先对每个用例文本描述，尤其是场景描述建立局部概念类图，再合并为整体概念类图。

  1. 根据用例文本描述识别候选类
    • 软件系统与外界交互时可能设计的对象与类
  2. 筛选候选类，确定概念类
    • 准则：依据系统需求；该类的对象实例的状态与行为是否完全必要
    • 候选类向概念类的转化：
      • 需要维护状态、表现行为：概念类
      • 需要维护状态，不需要表现行为：其他概念类的属性
      • 不需要维护状态，需要表现行为：审视需求，转交行为
      • 不需要维护状态、表现行为：剔除
  3. 识别关联
  4. 识别重要属性

3. 顺序图

也称交互图。

消息种类

  • 同步消息：实线，实心三角箭头
  • 异步消息：实线，>
  • 返回消息：虚线，>

顺序图的建立

  1. 确定上下文环境
  2. 根据用例描述找到交互对象
  3. 按照用例描述中的流程顺序逐步添加信息

4. 状态图

概念

  • 状态：描述系统在特定时间行为的可视化状况的集合
  • 状态转换：从一个状态到另一个状态
  • 事件：导致系统表现一些可预测的行为
  • 活动：作为状态转换的结果发生的过程

状态图的建立

  1. 确立上下文环境
  2. 识别状态
  3. 建立状态转换
  4. 完善状态图

五、需求文档化与验证

1. 用例文档

在用户的角度以用例文本为主描述软件系统与外界的交互。

  • 基本职责是把问题域信息和需求传达给软件系统解决方案的设计者

用例文档结构示例

  • 文档的信息
  • 用例图或用例列表
  • 用例描述

2. 软件需求规格说明文档

在软件产品的角度以系统级需求列表的方式描述软件系统解决方案。

用例 系统规格
侧重于交互流程 侧重于独立需求
基于一次交互 基于一次交互中的软件系统处理细节

需求规格说明文档结构示例

  • 引言
  • 总体描述
  • 详细需求描述
    • 对外接口需求
    • 功能需求
    • 性能需求
    • 约束
    • 质量属性
    • 其他需求
  • 附录
  • 索引

3. 文档化需求的注意事项

技术文档写作要点

  • 简洁
  • 精确
  • 易读
    • 有效使用引言、目录、索引等
    • 使用系统化的方式组织内容信息
      • 使用相同的语句格式来组织相似、关联的信息
      • 使用列表或表格来组织独立、并列的信息
      • 使用编号来表达繁杂信息之间的关系，包括顺序、嵌套、层次关系
  • 易修改
    • 独立性
    • ID
    • 引用，不重复

需求书写要点

  • 使用用户属于
  • 可验证
  • 可行性

需求规格说明文档书写要点

  • 充分利用标准文档模版，保持所有内容位置得当
  • 保持文档内的需求集具有完备性和一致性
  • 为需求划分优先级

4. 验证需求文档

方法：评审、开发系统测试用例、度量

开发系统测试用例

  1. 以需求为线索，开发测试用例套件
  2. 使用测试技术确定输入/输出数据，开发测试用例
