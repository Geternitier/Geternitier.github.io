软工2-07-需求分析

单纯的软件系统是不能解决问题的，只有和现实世界之间形成有效互动才能实现问题的解决。

07-需求分析

一、需求分析基础

需求分析的任务

  • 建立分析模型，达成开发者和用户对需求信息的共同理解
  • 依据共同的理解发挥创造性，创建软件系统解决方案

需求分析模型

模型是对事物的抽象，帮助人们在创建一个事物之前可以有更好的理解。

面向对象方法模型

模型 描述
用例图 描述用户与系统的交互。从交互的角度说明了系统的边界和功能范围。
类图 描述应用领域中重要的概念以及概念之间的关系。它捕获了系统的静态结构。
顺序图 描述系统中一次交互的行为过程，说明了在交互中的对象协作关系。
状态图 描述系统、用例或者对象在其整个生命期内的状态变化和行为过程。

结构化方法模型

模型 描述
数据流图 从数据传递和加工的角度，描述了系统从输入到输出的功能处理过程。运用功能分解的方法，用层次结构简化处理复杂的问题。
实体关系图 描述系统中的数据对象及其关系，定义了系统中使用、处理和产生的所有数据。

建模

建立模型的过程被称为建模。

  • 建模是对系统进行思考和推理的一种方式

  • 建模的目标是建立系统的一个表示，这个表示以精确一致的方式描述系统，使得系统的使用更加容易

  • 建模的常用手段：抽象和分解

二、面向对象分析

简单过程

  • 系统用例图——细化系统的对外交互——>用例模型/用例描述

  • 用例模型/用例描述——明确用例中的协作对象——>概念类图/领域模型

  • 用例模型/用例描述——明确用例中的协作行为——>顺序图、状态图、对象约束语言

1. 用例图

用例的定义

在系统（或者子系统或者类）和外部对象的交互当中所执行的行为序列的描述，包括各种不同的序列和错误的序列，它们能够联合提供一种有价值的服务。

用例图的基本元素

参与者、用例、关系、系统边界

  • 参与者（小人形）
    • 一个与正在开发的系统进行交互的用户或系统所扮演的角色
    • 用例图中的一个参与者可以代表多个用户或系统、扮演多种角色
  • 用例（椭圆）
  • 关系（不同的线和箭头及上面的补充文字）
  • 系统边界（矩形）
    • 参与者总在系统边界外

用例图的建立

  1. 目标分析与解决方向的确定
  2. 寻找参与者
  3. 寻找用例
  4. 细化用例
  • 如果用例的粒度不合适就需要进行细化和调整
    • 判断标准：用例描述了为应对一个业务事件，由一个用户发起，并在一个连续时间段内完成，可以增加业务价值的任务。

常见错误

  • 不要把用例细化为单个操作
    • 例：不要将用户管理细化为增加、修改和删除三个更小的用例，因为它们要联合起来才能体现出业务价值。
  • 不要把同一个业务目标细化为不同用例
    • 例：特价策略制定和赠送策略制定。
  • 不要把没有业务价值的内容作为用例
    • 例：“登录”(应该描述为安全性质量需求)、“数据验证”(应该描述为数据需求)、“连接数据库”(属性软件内部实现而不是需求)等。

2. 概念类图

概念类图又被称为领域模型。

类图是面向对象分析方法的核心，描述类和类之间的关系。

  • 和设计类图的不同：关注系统与外界的交互，而不是软件系统的内部构造机制
  • 类型、方法、可见性等复杂的软件构造细节不会在概念类图中

概念类图的基本元素

对象、类、链接、关联、继承

继承、组合、聚合、关联的图示：

classDiagram 
	A <|-- B
	C *-- D
	E o-- F
	G -- H

建立概念类图

先对每个用例文本描述，尤其是场景描述建立局部概念类图，再合并为整体概念类图。

  1. 根据用例文本描述识别候选类
    • 软件系统与外界交互时可能设计的对象与类
  2. 筛选候选类，确定概念类
    • 准则：依据系统需求；该类的对象实例的状态与行为是否完全必要
    • 候选类向概念类的转化：
      • 需要维护状态、表现行为：概念类
      • 需要维护状态，不需要表现行为：其他概念类的属性
      • 不需要维护状态，需要表现行为：审视需求，转交行为
      • 不需要维护状态、表现行为：剔除
  3. 识别关联
  4. 识别重要属性

3. 顺序图

也称交互图。

消息种类

  • 同步消息：实线，实心三角箭头
  • 异步消息：实线，>
  • 返回消息：虚线，>

顺序图的建立

  1. 确定上下文环境
  2. 根据用例描述找到交互对象
  3. 按照用例描述中的流程顺序逐步添加信息

4. 状态图

概念

  • 状态：描述系统在特定时间行为的可视化状况的集合
  • 状态转换：从一个状态到另一个状态
  • 事件：导致系统表现一些可预测的行为
  • 活动：作为状态转换的结果发生的过程

状态图的建立

  1. 确立上下文环境
  2. 识别状态
  3. 建立状态转换
  4. 完善状态图
