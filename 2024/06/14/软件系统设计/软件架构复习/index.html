1. 软件架构

1.1 什么是软件架构？

程序或计算系统的结构，由软件元素、这些元素外部可⻅的属性、这些元素的关系组成。

2. 质量属性

2.1 软件需求

功能需求

系统必须完成的、为涉众提供价值的内容。

  • 功能的实现可能⽤了很多的结构，但功能是与结构⽆关的。

质量需求（NFRs）

整个系统的期望特征，在功能需求之上。

  • 如果质量属性很重要，软件架构将约束功能的分配到不同的结构。

约束

0 ⾃由度、预定义的设计决定

  • 接受设计决定、协调其他受影响的设计决定，如此来满⾜约束

2.2 质量属性

内部/外部属性

建模质量属性场景

源、刺激、制品、环境、响应、度量

举例

可用性、互操作性、可修改性、性能、安全性、可测试性、其他

质量属性战术

战术是影响质量属性响应控制的设计决定。

  • 战术的集合被称为架构策略。

架构设计决策检查列表

3. 架构模式

3.1 架构模式

⼀系列在实践中能反复地看到的设计决定。

  • 上下文：经常出现的、普遍的、引起问题的情形
  • 问题：适当地概括的，在给定上下⽂中出现
  • 解决方案：成功的、适当抽象的架构解决⽅案

3.2 模块模式

分层模式（微内核模式）

3.3 组件-连接件模式

代理人模式、MVC模式、管道、过滤器模式、C/S模式、P2P模式、SO模式、发布-订阅模式、数据共享模式

3.4 分配模式

Map-Reduce模式、Multi-Tier模式

3.5 模式与战术

  1. 战术⽐模式简单：使⽤单⼀的结构或机制来处理单⼀的架构要求

  2. 模式通常将多个设计决定合并到⼀个包

  3. 模式和战术⼀起构成了软件架构师的主要⼯具

  4. 战术是设计的建造⽅块，⽤来创建架构模式

  5. ⼤多数模式由⼏个不同的战术组成

    • 所有战术为了⼀个共同⽬的

    • 经常被选择来保证不同的质量属性

4. 架构设计

4.1 通用设计策略

抽象、分解、迭代、重用

4.2 设计决策分类

职责、协调、数据、资源、元素映射、技术

4.3 属性驱动设计（ADD）

输⼊：需求

输出：软件元素、⻆⾊、职责、属性、关系

步骤：

  1. 确认需求
  2. 选择并分解元素
  3. 识别ASR
  4. 选择满足ASR的设计
    • 确认关注点
    • 列出可选项（模式/战术）
    • 选择模式/战术
    • 决定模式/战术和ASR间关系
    • 捕捉初步架构视图
    • 解决不一致
  5. 实例化元素，分配职责
  6. 定义元素接口
  7. 验证、细化需求
  8. 重复2-7直到所有架构重要需求被满足

5. 架构文档化

5.1 视图

样式、模式和视图

架构样式：元素和关系类型的专⻔化，以及如何使⽤它们的⼀组约束。

架构模式：软件系统中的基础结构组织⽅法的表达。

  • 架构模式关注问题和上下午，样式侧重方法。

样式的三个分类：

  • 一组样式单元：模块样式
  • 一组有运行时行为和交互的元素：组件-连接件样式
  • 与环境中的非软件结构关联：分配样式

架构视图：⼀组系统元素及其关系的表示。

  • 视图让我们将系统实体划分成感兴趣和易于管理的系统表示。
  • 不同的视图⽀持不同的⽬标和⽤户，凸显出不同系统元素和关系
  • 不同视图在不同程度上展现不同的质量属性

结构性视图

模块视图

模块是提供⼀套连贯的职责的实现单元。

包含：

  • 分解视图 Decomposition view
  • 使⽤视图 Uses view
  • 概括视图 Generalization view
  • 分层视图 Layered view
  • ⽅⾯视图 Aspects view
  • 数据模型视图 Data model view

元素：模块，提供⼀套连贯职责的实现单元

关系：

  • “⼀部分”：部分⼦模块和整体聚合模块之间的部分、整体关系

  • “依赖于”：两个模块之间的依赖关系。特定的模块视图详细说明了依赖关系的含义

  • “是”：更具体的⼦模块和更⼀般的⽗模块之间的泛化、专⻔化关系

约束：不同模块视图可能会施加特定的拓扑约束，例如限制模块之间的可⻅性。

⽤途

  • 代码构建蓝图

  • 变更影响分析

  • 规划增量开发

  • 需求追踪分析

组件-连接件视图

显示运行时组件。

  • 例如进程、对象、客户机、服务器和数据存储

  • “连接”指明了哪些连接器连接到哪些组件。

包含：

  • 管道-过滤器视图 Pipe-and-filter view
  • 客户机-服务端视图 Client-server view
  • 点对点视图 Peer-to-peer view
  • ⾯向服务架构视图 Service-oriented view
  • 发布-订阅视图 Publish-subscribe view
  • 共享数据视图 Shared-data view
  • 多层视图 Multi-tier view

元素：

  • 组件。主要处理单元和数据存储。组件有⼀组端⼝，通过这些端⼝与其他组件进⾏交互（通过连接器）

  • 连接器。组件间交互的途径。连接器有⼀组⻆⾊（接⼝），指示组件如何在交互中使⽤连接器

约束：

  • 组件只能连接到连接器，⽽不是直连其他组件

  • 连接器只能连接到组件，⽽不是直连其他连接器

  • 连接只能在相容的端⼝和⻆⾊上建⽴

  • 接⼝委托只能在两个相容端⼝或⻆⾊上定义

  • 连接器不能孤⽴出现，必须连接到组件

⽤途

  • 演示系统如何⼯作

  • 通过指定运⾏时元素的结构和⾏为来指导开发

  • 帮助解释运⾏时系统质量属性，如性能和可⽤性

分配视图

描述了软件单元到软件开发或执⾏环境元素的映射。

包含：

  • 部署视图 Deployment view
  • 安装视图 Install view
  • ⼯作安排视图 Work assignment view
  • 其他分配视图 Other allocation view

元素

  • 软件元素。软件元素具有环境所需的属性

  • 环境元素。环境元素具有提供给软件的属性

关系：被分配。软件元素被分配到环境元素，属性取决于特定视图。

⽤途：

  • ⽤于对性能、可⽤性、安全性 security 和安全性 safety 进⾏解释

  • ⽤于解释分布式开发和将⼯作分配给团队

  • ⽤于解释软件版本的并⾏访问

  • ⽤于解释系统安装的形式和机制

质量视图

包含

  • 安全视图 Security view

  • 性能视图 Performance view

  • 可靠性视图 Reliability view

  • 通信视图 Communication view

  • 异常视图 Exception view（错误处理视图 error- handling view）

6. 架构评价

6.1 架构分析+评价方法

软件架构分析⽅法 SAAM，Software Architecture Analysis Method

架构层级可修改性分析 ALMA，Architecture Level Modifiability Analysis

软件架构性能评估 PASA，Performance Assessment of Software Architecture

架构权衡分析⽅法 ATAM，Architecture Trade-off Analysis Method

6.2 ATAM

架构权衡分析⽅法，Architecture Tradeoff Analysis Method

  1. 合作与准备

    参与者：团队领导和主要项目决策者

    输入：架构文档

    输出：评价计划

  2. 评估1

    参与者：评估团队和项目决策者

    输出：架构展示、业务驱动因素、质量属性需求优先级列表、效用树、风险、敏感点和权衡点

    1. 展示 ATAM
    2. 展示业务驱动因素
    3. 展示架构
    4. 识别架构⽅法
    5. ⽣成效⽤树
    6. 分析架构⽅法

  3. 评估2

    参与者：评估团队、项目决策者、架构涉众

    输出：涉众社区的优先级场景列表、风险主题和受威胁的业务驱动因素

    1. 展示 ATAM 和结果
    2. 头脑⻛暴和定优先级
    3. 分析架构⽅法
    4. 展示结果

  4. 后续⾏动

    参与者：评估团队和主要涉众

    输出：最终评估报告

ATAM输出

  • 架构的简短展示
  • 业务⽬标的表达
  • 由质量属性场景表达的定优先级的质量属性需求
  • 效⽤树
  • ⼀组⻛险和⾮⻛险
  • ⼀组⻛险主题
  • 从架构决定到质量需求的映射
  • ⼀组确定的敏感点和权衡点
  • 最终评估报告
